## Часть A — Функциональная завершённость v1.0.x («все кнопки работают»)

**Цель:** каждый интерактивный элемент во всех 40 экранах (parent 12 · child 16 ·
specialist 7 · tutor 5) делает **осмысленное действие с реальными данными
устройства** — либо явно помечен «демо/скоро» без тупиков и 404.

**Что уже хорошо (аудит):** битых навигаций нет (все `navigate()` ведут на
существующие роуты), пустых `onClick` нет, `alert()` нет. Проблема — не «мёртвые»
кнопки, а **глубина**: демо-данные вместо реальной ленты и действия без эффекта.

### A.0 — Методология (Задача 1: аудит-реестр кнопок)
Пройти все 40 экранов, для каждого — таблица:

| Экран | Элемент | Ожидание | Факт | Вердикт |
|---|---|---|---|---|

Вердикты и что делать:
- **OK** — работает с реальными данными → ничего.
- **DEMO-DATA** — экран/действие на `DEMO_CHILDREN`, а не на реальной ленте →
  привязать к `useEventStore`/реальному ребёнку (см. A.2).
- **NO-EFFECT** — обработчик есть, но ничего не сохраняет/не меняет состояние →
  дописать до localStorage/стора.
- **STUB** — «появится в следующей версии» → реализовать или честно скрыть кнопку
  (не показывать нерабочее семье).
- **DEAD** — тупик/битая ссылка → починить навигацию.

Результат аудита — `docs/BUTTON_AUDIT.md` (реестр + вердикты). Это вход для A.1–A.4.

### A.1 — Известные точечные цели (найдено при аудите)
1. **Reports (specialist):** `Скачивание PDF` и `Отправка` = `showToast('появится
   в следующей версии')` ([Reports.tsx:111,119](../apps/prototype/src/pages/specialist/Reports.tsx)).
   → Реализовать: PDF через `window.print()`/`jsPDF` из реальной ленты, или скрыть
   до v3. Не показывать нерабочую кнопку на пилоте.
2. **Демо-гид на 18 экранов** наезжает поверх живых экранов (z-index/условие
   показа) — баг, замеченный на устройстве. → Условие запуска тура + слой/оверлей.
3. **`useRealtimeEvents`** — socket.io к stateless-прокси, который сокетов не
   имеет: мёртвый код, всегда падает в offline. → Удалить хук и его вызовы
   (закрывает и лишние 404-подключения).
4. **404-эндпоинты** `/api/events`, `/api/recordings`, `/api/children` — стор
   штатно откатывается на localStorage, но нужно **убедиться, что все CRUD-действия
   100% работают локально** (создание/правка/удаление события сохраняются и
   переживают перезапуск). Убрать сетевые попытки к несуществующим эндпоинтам,
   чтобы не мигали 404 и не тормозили UI.

### A.2 — По ролям (scope завершённости)
- **Parent (12 экранов):** голос ✅. Проверить: EventTimeline/EventDetails на
  **реальной** ленте (не демо); правка/удаление события; CareDiary, Analytics,
  Behavior, Assistant — считают из реальных событий устройства (пусто → честный
  empty-state, не фейк-графики); Profile — имя ребёнка/сброс данных/экспорт.
- **Child (16 экранов):** **AAC-карточки (ChildCards, PhraseBuilder, ChildSpeak,
  NowNext) должны логировать `communication`-событие в ленту + озвучивать (TTS)** —
  сейчас во многом визуал. «Позвать маму» — реальное уведомление/запись события.
  CalmMode — без принуждающих таймеров (по STRATEGY §4.8). Это и есть «кнопки
  срабатывают» для детского UI.
- **Specialist (7):** ABC, CarePatterns, CommunicationProfile, Reports,
  SupportPlan, Dashboard — из **реальной** ленты; экспорт (A.1.1); формулировки
  без medical claims (уже проверено, держать).
- **Tutor (5):** Home, Report, AIReview, ChildProfile — те же принципы; отчёты
  из реальных данных.

### A.3 — Критерии приёмки Части A
- [ ] `docs/BUTTON_AUDIT.md`: 100% интерактивных элементов с вердиктом, 0 в статусе DEAD.
- [ ] Ни одной кнопки со скрытой заглушкой на видимых пилоту экранах (STUB → реализован или скрыт).
- [ ] Все CRUD события/записи сохраняются локально и переживают перезапуск APK.
- [ ] Child AAC логирует `communication`-события + TTS.
- [ ] Нет обращений к несуществующим эндпоинтам (нет мигающих 404).
- [ ] Прогон на устройстве по [QA_PLAN_V1.md](QA_PLAN_V1.md): каждый экран без тупиков.

### A.4 — Гигиена (в ту же волну)
- Удалить мёртвый realtime/socket код; сверить `docs/CURRENT_ARCHITECTURE.md`
  (аудио-контракт получил `aiFallback`/`aiError`, STT — детект формата).
- Контрактный тест схемы `/api/audio/ingest` (граница фронт↔бэк, STRATEGY §6.7).

## Часть B — Стратегический роадмап (из STRATEGY.md §5)

> Правило волн: **после Wave 0/1 — фидбек, потом фичи.** Часть B начинается
> только когда Часть A закрыта и пилот подтвердил, что голосом реально пользуются.

### B1 · v1.5 — «данные превращаются в смысл»
- Формализовать `QoldauEvent` (schemaVersion, occurredAt/recordedAt, source, abc,
  sensoryContext) + интерфейс `EventStorage` (STRATEGY §6.2–6.3).
- Паттерн-движок **на устройстве** поверх реальной ленты: scatterplot «день×время»,
  топ-триггеры, серии.
- Авто-ABC + sensory из голосового рассказа (расширение RU-промпта + поля события).
- Недельный дайджест (локальная статистика → опц. LLM-формулировка).
- Golden set 100+ RU-фраз + eval-harness в `apps/api` (STRATEGY §7.3) — до масштаба.

### B2 · v2.0 — «приложение учит и помогает»
- Just-in-time микрообучение (карточки навыков по CST, привязка к событиям ленты).
- Конструктор визуальных рутин (Now/Next → полные расписания).
- ATEC-трекинг динамики (родительская шкала, график).
- **KK-локализация** (обещание бренда).
- Хранилище → SQLite (`@capacitor-community/sqlite`) через `EventStorage`.

### B3 · v2.x — коммуникация и семья
- AAC-ядро: core-словарь RU/KK, фиксированная сетка, TTS, `communication`-события.
- Социальные истории (LLM-ассистент, опция).
- Аккаунты + синхронизация (append-only лог + LWW; magic-link) — STRATEGY §6.4.
- Экспорт PDF-отчёта для специалиста/ПМПК (доводит A.1.1 до продакшена).

### B4 · v3.x — специалист и экосистема
- Кабинет специалиста (доступ по приглашению, GAS-цели, ABC-аналитика, отчёты).
- B2B мульти-семейная панель для центров.
- Weekly AI-дайджест поверх всей истории.
- iOS.

## Процесс и последовательность

**Связка агентов** (STRATEGY §8, проверено 0 конфликтов): Codex=`apps/api`,
MiniMax=`apps/prototype`, Claude=`docs`/интеграция/релизы. Contract-first,
один агент = одна ветка = одна зона, интеграция `--no-ff` Claude'ом с прогоном
обоих тест-сьютов.

**Порядок:**
1. **Сейчас:** финальный on-device QA Части A-базы (голос уже работает) → снять
   Draft с [PR #2](https://github.com/Krimskii/qoldau/pull/2), решить расхождение
   master↔integration, merge.
2. **Часть A** (v1.0.x): Задача 1 (аудит-реестр) → распараллелить A.1–A.4 по ролям
   между MiniMax (UI/действия) и Codex (если нужны серверные поля). Claude ведёт
   `BUTTON_AUDIT.md` и интеграцию.
3. **Wave 0 → фидбек** (1 семья) → правки → Wave 1 (≈20 семей).
4. **Часть B** — по волнам, строго после подтверждённого использования.

**Метрики-ориентиры** (STRATEGY §9): активация >70%, наблюдений/семью/неделю ≥5,
доля голоса >50%, time-to-log <40с, aiFallback <5%.
