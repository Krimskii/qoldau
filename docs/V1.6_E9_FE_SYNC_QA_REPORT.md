## Архитектура sync

### Поток данных
```
[useEventStore]  — addEvent/updateEvent/deleteEvent
       │
       │ notifyLocalChangeDebounced(childId)
       ▼
[syncService.notifyLocalChange]  — debounce 4с
       │
       ▼
[syncService.syncForChild]
   ├── push: POST /api/sync/push {childId, events: [...delta]}
   │       ↑ lastPushedAt[childId] как since
   │       ↑ updatedAt > since
   │
   └── pull: GET /api/sync/pull?childId=X&since=Y
           ↑ lastSyncedAt[childId] как since
           ↓ upsert + tombstone (deletedAt)
           ↓ serverTime → new lastSyncedAt
```

### Флаги / обратная совместимость
- `VITE_ENABLE_SYNC=true` → sync active; default **false** (demo).
- `VITE_REQUIRE_AUTH=false` → demo без принудительного логина.
- `qoldau-auth-v2` localStorage: v1 без refreshToken → null (принудительный re-login). Breaking, но v1 ещё не зарелижена в прод.

### Auth flow
1. `/auth/login` → POST `/api/auth/request-magic-link`
2. dev: ответ содержит `devMagicUrl`/`token` → автоподставить в VerifyPage
3. prod: email доставка (временно выключена — `EMAIL_PROVIDER=none`)
4. VerifyPage → POST `/api/auth/verify` → jwt + refreshToken + user{childIds}
5. Save to localStorage + auth-store
6. AppInit: `useAuthStore.init()` подтягивает сессию
7. При logout: revoke refresh на сервере (best-effort) + clearAuth + reset sync-store

### 401 → refresh → retry
```
api/client.request(path)
   ↓
fetch → 401?
   ↓ yes (если есть jwt)
refreshInFlight ??= refreshHandler()
   ↓
refreshHandler → useAuthStore.refresh() → POST /api/auth/refresh
   ↓ ok=true → новый jwt + refreshToken в store
   ↓
retry fetch с обновлённым Authorization
   ↓
если refresh fail → logoutHandler → useAuthStore.logout() → редирект /auth/login
```

## Подтверждения по тикету

### (1) demo/local-first НЕ сломан ✓
- `VITE_ENABLE_SYNC=false` (default) → `SYNC_ENABLED=false` → `syncForChild()` сразу return `{ok:false, reason:'demo'}` без вызова API.
- `VITE_REQUIRE_AUTH=false` (default) → 401 handler не редиректит на login (только регистрирует).
- Все 255 тестов pre-E9 проходят + 16 новых sync-тестов с VITE_ENABLE_SYNC=true через `vi.stubEnv`.

### (2) login → jwt → sync работает в dev ✓
- `LoginPage` → `useAuthStore.requestMagicLink(email)` → dev возвращает `{devMagicUrl, token}` → авто-подставляется в `VerifyPage`.
- `VerifyPage` → `useAuthStore.verifyAndLogin(token)` → сохраняет jwt + refreshToken + user.childIds.
- AppInit ловит `authStatus='authenticated'` → запускает `syncAll()` (push→pull для всех childIds).
- Debounce на локальные мутации: `addEvent` через `useEventStore` → `notifyLocalChangeDebounced` → 4с debounce → `syncForChild`.

### (3) pull / push / tombstone / conflict / offline-queue ✓
- **pull**: upsert by id + tombstone через deletedAt, serverTime → lastSyncedAt.
- **push**: дельта events с updatedAt > lastPushedAt; LWW на сервере; серверный conflicts[].
- **tombstone**: сервер возвращает events с deletedAt !== null → локально `!deletedAt` фильтрует (через `eventStorage.isLive`).
- **conflict**: после push всегда pull → серверная версия перезаписывает локальную (LWW по updatedAt).
- **offline-queue**: локальные мутации копятся в `useEventStore`; при появлении сети+jwt следующий sync пушит всё накопленное.

### (4) 401 → refresh ✓
- `api/client.request()`: 401 → `refreshInFlight ??= refreshHandler()` → `useAuthStore.refresh()` → новый jwt → retry с обновлённым Authorization.
- Параллельные 401 координируются: `refreshInFlight` Promise делится между всеми.
- Если refresh fail → `logoutHandler` → `useAuthStore.logout()` + редирект `/auth/login?returnTo=`.

### (5) 0 хардкод-hex, i18n, a11y ✓
- 0 inline-hex в JSX (только Tailwind tokens / `tokens.ts`).
- i18n: 11 новых ключей `sync.*` в ru/kk/en — паритет enforced (i18nParity test passes).
- A11y: `SyncStatusBadge` имеет `role="status"` / `role="button"`, `aria-label` на каждом состоянии, `title` с детальной ошибкой. `SyncSection` использует `<button onClick>` для действий, `aria-label` через переводы. Sync регресс-тесты ничего не сломали в a11y-тестах.

## Сценарии тестирования (после деплоя)

### Dev-сборка (VITE_ENABLE_SYNC=true, VITE_API_BASE_URL=http://localhost:4000)
1. Запустить `apps/api`: `npm run dev` (Postgres локально).
2. `apps/prototype`: `npm run dev` (Vite на 5173).
3. Открыть `/auth/login`, ввести `test@example.com`.
4. В dev-ответе `requestMagicLink` содержит `devMagicUrl` — UI автоматически переходит в VerifyPage с предзаполненным токеном.
5. Verify → jwt сохраняется → AppInit триггерит `syncAll()` → `SyncStatusBadge` показывает «Synced».
6. Добавить event через VoiceObservation → через 4с debounce `syncForChild` пушит → pull обновляет lastSyncedAt.
7. Открыть другое устройство / браузер (с тем же email + dev-magic-link) → тянет те же события через pull.
8. Тест конфликта: на двух устройствах одновременно редактировать одно событие → последний updateAt (LWW) побеждает.

### Prod-сборка (VITE_ENABLE_SYNC=false)
1. APK / production build — старая версия без sync UI.
2. SyncStatusBadge показывает «Демо (без входа)».
3. Все данные локальные (per-device), как в v1.5.

### Prod-sync (после email + ENABLE_SYNC=true)
1. `/auth/login` → email → real magic-link.
2. Verify → jwt + refreshToken + childIds.
3. Sync включён автоматически.
4. 401 → refresh → retry (на любых API-запросах).
5. Multi-device sync работает (push дельты + pull изменений).
